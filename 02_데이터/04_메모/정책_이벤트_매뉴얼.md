## 1. 정책 분류 체계

### 1.1 정책 유형 (4대 분류)

```
정책유형
├── 대출규제/완화
│   ├── LTV (Loan to Value)
│   ├── DTI (Debt to Income)
│   ├── DSR (Debt Service Ratio)
│   └── 대출한도
│
├── 세금규제/완화
│   ├── 취득세
│   ├── 양도세
│   ├── 종합부동산세
│   └── 재산세
│
├── 거래규제/완화
│   ├── 전매제한
│   ├── 청약제도
│   ├── 분양가상한제
│   └── 실거주의무
│
└── 공급규제/완화
    ├── 재건축 초과이익환수
    ├── 재개발 안전진단
    ├── 용적률 규제
    └── 공급확대 정책
```

---

### 1.2 규제/완화 구분

| 방향 | 설명 | 투자수요 영향 | 부호 |
|------|------|--------------|------|
| **규제 (Tightening)** | 거래·대출·세금 제한 강화 | 하락 ↓ | 음수 (-) |
| **완화 (Easing)** | 거래·대출·세금 제한 완화 | 상승 ↑ | 양수 (+) |

## 2. 강도 점수 기준

### 2.1 1~5점 척도

| 점수 | 규제 방향 | 완화 방향 | 시장 충격 | 예시 |
|------|----------|----------|----------|------|
| **1** | 경미한 규제 | 경미한 완화 | 최소 | 특정 지역 미세 조정 |
| **2** | 소규모 규제 | 소규모 완화 | 작음 | 1-2개 지역 규제 추가 |
| **3** | 중간 강도 | 중간 강도 | 중간 | 수도권 일부 규제 |
| **4** | 강력한 규제 | 상당한 완화 | 큼 | 전국 주요 지역 규제 |
| **5** | 초강력 (시장충격) | 전면적 완화 | 최대 | 전국 동시 다발 규제 |

---

### 2.2 강도 점수 산정 기준

#### 가중 요소

| 요소 | 가중치 | 설명 |
|------|--------|------|
| **영향 범위** | 40% | 전국 > 수도권 > 특정 지역 |
| **규제 강도** | 30% | LTV 20%p 하락 > 10%p 하락 |
| **정책 개수** | 20% | 대출+세금+거래 동시 > 단일 |
| **시장 반응** | 10% | 뉴스 빈도, 검색량 급증 |

#### 계산식

```
강도점수 = (
    영향범위점수 × 0.4 +
    규제강도점수 × 0.3 +
    정책개수점수 × 0.2 +
    시장반응점수 × 0.1
)

반올림하여 1~5점
```

#### 예시: 8.2 대책

```
영향범위: 전국 15개 지역 → 4점
규제강도: LTV 70%→40% (30%p 하락) → 5점
정책개수: 대출+세금+거래 3개 → 5점
시장반응: 뉴스 급증, 패닉 → 5점

강도점수 = 4×0.4 + 5×0.3 + 5×0.2 + 5×0.1
        = 1.6 + 1.5 + 1.0 + 0.5
        = 4.6 → 반올림 5점
```

## 5. 정책 DB 스키마 (현행)

### 5.1 운영 테이블 정의 (`정책_이벤트`)

현재 코드(`collectors/정책_웹검색.py`)는 `정책_이벤트` 테이블을 사용합니다.

```sql
CREATE TABLE IF NOT EXISTS 정책_이벤트 (
    id INTEGER PRIMARY KEY,
    일자 DATE,
    정책_유형 VARCHAR,         -- 예: 대출규제, 세제, 청약, 재개발, 분양, 기타
    규제_방향 VARCHAR,         -- 강화, 완화, 유지
    규제_강도 INTEGER,         -- -5 ~ +5
    주체 VARCHAR,              -- 발표 기관
    핵심_내용 TEXT,
    영향_지역 VARCHAR,         -- 전국/수도권/서울 등
    시행일 DATE,
    주요_변경사항 JSON,
    검색쿼리 TEXT,
    수집방법 VARCHAR,          -- 현재: 웹검색
    수집일시 TIMESTAMP,
    원본_검색결과 TEXT
);
```

### 5.3 기존 `정책` 테이블과의 관계

- 본 문서의 과거 정책 서술(3장)은 레퍼런스 데이터셋이다.
- 운영 적재는 `정책_이벤트`를 기준으로 하며, 필요 시 `정책` 뷰를 별도로 생성한다.

### 6.4 데이터 품질 점검

```sql
-- 최근 30일 수집 건수
SELECT COUNT(*) FROM 정책_이벤트
WHERE 일자 >= CURRENT_DATE - INTERVAL 30 DAY;

-- 중복 후보 (같은 일자 + 같은 핵심 내용)
SELECT 일자, 핵심_내용, COUNT(*) AS cnt
FROM 정책_이벤트
GROUP BY 1, 2
HAVING COUNT(*) > 1
ORDER BY cnt DESC;
```
