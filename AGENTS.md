# APT-Price-Pattern Workspace Guide

## 목적
- 이 저장소는 일반 애플리케이션 코드베이스보다 `DuckDB 기반 부동산 분석 워크플로우`를 관리하는 작업 공간에 가깝다.
- 최종 지향점은 `시계열 시장 분석 + 지역/생활권 분석 + 단지 상품성/입지 분석 + 임차/정책/공급 해석 + 확률적 후보 선별`을 결합한 `투자 멘토 시스템`이다.
- 에이전트는 임의 규칙을 새로 만들기보다, 기존 기준 문서와 참조 데이터를 현재 작업 단위로 실행 가능하게 옮기는 역할에 집중한다.

## 작업 원칙
- 시작 전에 DuckDB 접근 가능 여부, 스키마, 테이블, 주요 컬럼, 최신 계약년월을 먼저 확인한다.
- `execution_guide.md`가 바뀌었거나 새 참조 데이터가 추가됐으면, 전체 재점검 대신 `경량 점검 -> 전처리 재준비 -> 해당 분석 단계 재실행` 순서로 간다.
- 문서상 컬럼명과 실제 DB 컬럼명이 다를 수 있으므로, 의미 기준으로 매핑한 뒤 작업한다.
- 매매/전세/월세는 분리 분석하고, 전세와 월세를 직접 환산 비교하지 않는다.
- 매매 대표가격만으로 계산한 값은 `수익률`이 아니라 `가격상승률`, `가격하락률`, `가격변화율`로 표기한다.
- 투자금 개념이 들어가는 경우에만 `갭투자 수익률`처럼 별도 용어를 쓴다.
- 결론은 항상 `사실 / 해석 / 가설`을 구분해서 정리한다.
- 데이터로 확인되지 않은 부동산 통념은 가설 또는 참고 배경으로만 다룬다.
- 수도권 심화 분석과 지방 보조 분석을 섞지 않는다.
- 교통축 기반 해석은 `02_데이터/02_참조/교통축_인접_매핑.csv`를 우선 사용한다.
- 토지거래허가구역은 DuckDB `토지거래허가구역`을 허가 승인/수집 이력 보조 데이터로 보고, 현재 활성 구역 자체는 최신 공식 웹 확인 결과와 수동 참조표로 보완한다.
- 외부 프로젝트의 결론을 그대로 가져오지 말고, 현재 프로젝트 데이터와 기준 문서에 맞춰 새로 검증한다.

## 상품성/입지 원칙
- 좌표는 DuckDB 내부 소스를 우선 사용한다.
  - 1순위: `main.좌표`, `공동주택_서울.좌표X/좌표Y` 등 내부 좌표
  - 2순위: `02_데이터/03_가공/상품성/단지_좌표.csv`
- 좌표/입지 보강 시 항상 `좌표 출처`, `조인 키`, `매칭률`, `적용 지역`, `누락 지역`을 남긴다.
- `02_데이터/03_가공/상품성/아파트_단지_위치변수.csv`는 서울/경기/인천 `19,326개 단지` 기준 심화 데이터이므로 전국 공통 변수처럼 쓰지 않는다.
- 전국 공통 분석의 기본 상품성 축은 `연식`, `세대수`, `동수`, `면적 구성`, `건축년도/사용승인연도`, `거래량`, `가격 수준`이다.
- 수도권 심화 분석에서는 필요 시 `역세권`, `학교/학원`, `상권`, `공원`, `한강`, `CBD`, `정비사업`을 함께 사용한다.
- 서울/핵심 수도권의 노후 단지는 단순 감가상각으로만 읽지 않는다. `재건축 기대`, `토지 희소성`, `한강/CBD/핵심 생활권 접근성`, `급지`, `절대가격 부담`과 함께 해석한다.
- 지방광역시와 주요중소도시는 수도권 변수 체계를 그대로 복제하지 말고, 자차 생활권, 산업, 행정, 도심, 신도시 축을 우선 해석한다.

## 실행 방식
- 큰 작업은 한 번에 끝내려 하지 말고 체크리스트 번호 단위로 끊어서 진행한다.
- `AGENTS.md`는 작업 환경과 장기 기준을 설명하는 문서이고, `RULES.md`는 현재 세션에서 바로 따라야 할 실행 규칙 요약본이다.
- 새 참조 데이터가 추가됐을 때 기본 순서는:
  1. `1번 경량 점검`
  2. `2번 전처리 및 분석 테이블 준비`
  3. 영향 받는 분석 단계 재실행
- 좌표/상품성/입지 데이터가 추가되면, 곧바로 보고서만 수정하지 말고 먼저 `전처리/분석 테이블`을 다시 준비한다.
- 각 단계 종료 시 `완료 항목 / 미완료 항목 / 이슈 / 생성·수정 파일` 형식으로 정리한다.
- 범위를 벗어난 수정은 하지 않는다.
