# Chapter 4. 왜 DuckDB인가

## 이 장에서 독자가 얻는 것

- 이 프로젝트가 왜 DuckDB를 중심 저장소로 택했는지 이해하게 된다.
- 엑셀, 판다스, 서버형 데이터베이스와 DuckDB의 역할 차이를 구체적으로 비교할 수 있게 된다.
- `저장`, `조회`, `스크립트`, `보고서`, `노트북`이 어떻게 하나의 흐름으로 연결되는지 알게 된다.

## 왜 이 장이 필요한가

부동산 데이터를 어느 도구 위에 올릴지는 단순한 취향 문제가 아니다. 도구 선택은 곧 작업 방식의 선택이고, 작업 방식의 선택은 결국 해석 습관을 바꾼다.

엑셀은 빠르게 훑어보기에 좋지만 반복성과 추적성이 약하다. 판다스는 유연하지만, 데이터가 커질수록 중간 상태 관리와 재현성이 흔들리기 쉽다. 서버형 데이터베이스는 강력하지만 개인 분석 환경에서는 부담이 커질 수 있다. 반면 DuckDB는 파일 기반이면서도 SQL로 큰 데이터를 빠르게 다룰 수 있고, 파이썬 스크립트와 노트북, 보고서 생성을 한 흐름으로 묶기 좋다.

이 장의 목적은 `요즘 많이 쓰니까 DuckDB를 쓴다`가 아니라, 왜 이 프로젝트의 목표와 DuckDB의 성격이 잘 맞는지를 분명히 하는 데 있다.

## 1. 이 프로젝트가 요구하는 도구 조건

이 프로젝트는 단순 조회보다 더 복합적인 요구를 가진다.

### 1.1 데이터가 여러 층으로 나뉜다

한 번에 다뤄야 하는 데이터는 아래처럼 다양하다.

- 매매 실거래
- 전세·월세 데이터
- 공동주택 마스터
- 정책 자료
- 입주예정 물량
- 미분양
- 좌표와 입지 변수
- 호가와 매물 데이터

즉 단일 CSV를 열어보는 수준의 도구로는 부족하다.

### 1.2 반복 조회가 많다

같은 데이터를 한 번 보고 끝내는 것이 아니라, 다음과 같은 반복 작업이 많다.

- 최신 계약년월 확인
- 특정 권역만 필터링
- 면적 구간별 대표가격 계산
- 전세가율 계산
- 미분양과 입주예정 물량 결합
- 상품성 변수와 거래 데이터 조인
- 후보 지역 좁히기

이 반복성은 `조회 언어`와 `재현성`을 요구한다.

### 1.3 결과물이 보고서와 노트북으로 갈라진다

이 프로젝트는 최종 결과가 하나가 아니다.

- 보고서
- CSV 보조표
- 예측 결과표
- 노트북 검증 차트

즉 같은 데이터 기반에서 서술용 산출물과 검증용 산출물을 동시에 만들어야 한다.

## 2. 왜 엑셀만으로는 부족한가

엑셀은 여전히 훌륭한 도구다. 빠르게 훑어보고, 표를 보고, 간단한 계산을 하기에 좋다. 하지만 이 프로젝트 수준에서는 몇 가지 한계가 분명해진다.

### 2.1 반복 작업이 누적되면 추적이 어렵다

같은 필터와 계산을 여러 번 반복할수록, 누가 어떤 기준으로 시트를 바꿨는지 추적이 어려워진다.

### 2.2 큰 데이터와 다단 조인에 약하다

거래 데이터, 공동주택 마스터, 입지 변수, 공급 데이터, 정책 참조표를 여러 키로 조인하는 순간 관리 비용이 크게 올라간다.

### 2.3 문서화가 약하다

엑셀 안에서도 기준을 기록할 수는 있지만, `왜 이 계산을 했는가`, `어떤 예외를 제외했는가`, `다음 달에도 어떻게 반복할 것인가`를 일관되게 남기기는 어렵다.

즉 엑셀은 관찰 도구로는 좋지만, 이 프로젝트가 원하는 `분석 체계의 중심 저장소`로는 약하다.

## 3. 왜 판다스만으로도 부족한가

판다스는 매우 강력하다. 실제로 이 프로젝트도 파이썬 스크립트와 함께 사용한다. 하지만 판다스만으로 전체 시스템을 운영하면 몇 가지 문제가 생긴다.

### 3.1 중간 상태가 코드 밖으로 흩어지기 쉽다

노트북 셀을 여러 번 실행하다 보면 현재 DataFrame이 어떤 필터와 조인을 거친 상태인지 흐려지기 쉽다.

### 3.2 SQL처럼 구조적 조회를 공유하기 어렵다

복잡한 조건을 여러 번 재사용할 때, SQL 뷰처럼 명시적으로 이름 붙여 관리하는 방식이 더 읽기 쉽고 공유하기 쉽다.

### 3.3 메모리 의존성이 커질 수 있다

판다스는 유연하지만, 큰 데이터를 다룰 때는 메모리 부담과 처리 시간 관리가 필요하다. 반면 DuckDB는 파일 기반 질의와 집계에 강하다.

즉 판다스는 보조 엔진으로 훌륭하지만, 이 프로젝트에서는 DuckDB와 함께 쓰일 때 더 강하다.

## 4. DuckDB가 이 프로젝트와 잘 맞는 이유

### 4.1 파일 하나로 중심 저장소를 유지할 수 있다

이 프로젝트의 핵심 DB는 외부 경로의 DuckDB 파일 하나에 모여 있다.

- `/Volumes/T9/duckdb-analytics/db/apartment.duckdb`

서버를 운영하지 않아도 되고, 분석용 DB를 독립적으로 관리할 수 있다. 개인 연구, 반복 보고서, 노트북 검증 흐름에 잘 맞는다.

### 4.2 SQL 뷰 중심으로 분석 단계를 남길 수 있다

이 프로젝트는 단순 쿼리 실행이 아니라 `분석 테이블과 뷰 준비`를 중요하게 본다. DuckDB는 이 과정에 잘 맞는다.

예를 들어 아래 같은 흐름을 자연스럽게 만들 수 있다.

- 매매 월간 대표가격 뷰
- 전세 월간 대표가격 뷰
- 전세가율 뷰
- 입주예정 물량 집계 뷰
- 호가-실거래 비교 뷰
- 토지거래허가 보조 뷰

즉 데이터 준비 과정 자체를 구조로 남길 수 있다.

### 4.3 파이썬과 연결이 쉽다

DuckDB는 파이썬에서 바로 연결해 쿼리를 실행하고 DataFrame으로 받을 수 있다. 그래서 다음과 같은 흐름이 자연스럽다.

1. DuckDB에서 데이터 정리
2. 파이썬 스크립트에서 집계 결과 호출
3. 보고서 Markdown 생성
4. 노트북에서 차트 검증

이 연결성이 이 프로젝트에 중요하다.

### 4.4 원천 파일 형식이 섞여 있어도 흡수하기 좋다

이 프로젝트는 `utf-8-sig + |`, `cp949 + ,`, `xls/xlsx` 등 형식이 섞인 원천을 다룬다. DuckDB는 CSV, Parquet, 외부 파일 조회와 임시 변환에 유연한 편이라, 중간 정리 단계의 중심 엔진으로 쓰기 좋다.

## 5. DuckDB를 쓴다고 해서 자동으로 좋아지지는 않는다

중요한 점이 있다. DuckDB는 강력하지만, 잘못 쓰면 그냥 빠른 저장소일 뿐이다.

### 5.1 스키마와 뷰 설계가 필요하다

무엇을 어떤 이름의 뷰로 남길지, 어떤 조인 규칙을 기준으로 삼을지 정하지 않으면 DuckDB도 결국 쿼리 메모장으로 전락한다.

### 5.2 기준 문서가 없으면 도구만 남는다

이 프로젝트는 `prompt_final.md`, `checklist_execution.md`, `execution_guide.md` 같은 기준 문서가 먼저 있고, DuckDB는 그것을 실행하는 엔진 역할을 맡는다.

### 5.3 도구가 해석을 대신하지 않는다

DuckDB가 빠르게 집계를 해준다고 해서 그 결과가 곧바로 좋은 투자 판단이 되지는 않는다. 해석은 여전히 별도의 작업이다.

## 6. 이 프로젝트의 권장 분업 구조

이 프로젝트는 대략 아래처럼 분업한다.

### 6.1 DuckDB

- 중심 저장소
- 대규모 조회
- 집계
- 뷰 생성
- 기준 테이블 유지

### 6.2 파이썬 스크립트

- 반복 실행
- 리포트 초안 생성
- CSV 산출
- 점수 계산

### 6.3 Markdown 문서

- 기준 문서
- 보고서
- 집필 원고
- 방법론 설명

### 6.4 노트북

- 차트 검증
- 특정 후보 drill-down
- 보고서 수치 재확인

이 분업 구조가 중요하다. 하나의 도구로 모든 것을 해결하려고 하면 오히려 관리가 어려워진다.

## 7. 독자가 자신의 환경에서 DuckDB를 도입할 때 체크할 것

### 7.1 DB 파일을 어디에 둘 것인가

저장소 안에 둘지, 외부 경로에 둘지 먼저 정해야 한다. 데이터가 커질수록 코드와 DB를 분리하는 편이 관리에 유리하다.

### 7.2 어떤 뷰를 고정 자산으로 만들 것인가

매번 다시 계산하지 않고 이름 붙여 유지할 핵심 뷰를 정해야 한다.

예:

- 매매 대표가격
- 전세 대표가격
- 전세가율
- 입주예정 물량 집계
- 미분양 요약

### 7.3 SQL과 파이썬의 경계를 어디에 둘 것인가

무엇을 DB 안에서 처리하고, 무엇을 파이썬에서 처리할지 나눠야 한다. 일반적으로 집계와 정리는 DB에, 최종 서술과 포맷 생성은 파이썬에 두는 편이 안정적이다.

## 해석 예시

- 사실:
  이 프로젝트는 핵심 데이터를 DuckDB 파일에 두고, 저장소 안에는 기준 문서, 스크립트, 결과물을 분리해서 관리한다.
- 해석:
  이런 구조는 개인 연구 환경에서도 반복성과 재현성을 확보하기 좋다.
- 가설:
  독자가 엑셀 중심 작업에서 DuckDB 중심 구조로 옮기면, 한 번 쓴 분석을 다시 돌리는 비용이 크게 줄어들 가능성이 높다.

## 흔한 오해

- `DuckDB를 쓰면 자동화가 완성된다`
  도구가 아니라 설계가 자동화의 핵심이다.
- `DuckDB는 개발자용이라 투자자에게는 과하다`
  오히려 반복 분석이 많아질수록 개인 투자자에게 더 유리할 수 있다.
- `SQL을 잘 모르면 시작할 수 없다`
  깊은 최적화는 나중 문제다. 먼저는 반복 가능한 조회 구조를 만드는 것이 중요하다.

## 직접 해볼 과제

1. 지금 자신의 작업에서 엑셀, 판다스, 데이터베이스가 각각 어떤 역할을 하고 있는지 적어본다.
2. 반복해서 다시 만드는 계산 세 가지를 골라본다.
3. 그 계산 중 무엇을 DuckDB 뷰로 고정하면 좋은지 적어본다.
4. 보고서용 작업과 검증용 작업이 현재 어떻게 섞여 있는지 점검해본다.

## 체크포인트

- 나는 도구를 취향이 아니라 작업 구조 기준으로 선택하고 있는가
- 나는 데이터를 읽는 단계와 결과를 쓰는 단계를 분리하고 있는가
- 나는 같은 집계를 다음 달에도 다시 돌릴 수 있는 구조를 갖고 있는가
