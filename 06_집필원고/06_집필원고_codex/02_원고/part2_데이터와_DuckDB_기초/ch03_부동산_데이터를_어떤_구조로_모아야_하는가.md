# Chapter 3. 부동산 데이터를 어떤 구조로 모아야 하는가

## 이 장에서 독자가 얻는 것

- 부동산 데이터가 많다고 해서 바로 분석 가능한 상태가 되는 것은 아니라는 점을 이해하게 된다.
- `거래`, `단지 마스터`, `정책`, `공급`, `상품성/입지`, `보조시장 데이터`를 어떤 역할로 분리해야 하는지 알게 된다.
- 이 책 전체에서 왜 데이터의 양보다 `데이터 계층 구조`를 먼저 세우는지 납득하게 된다.

## 왜 이 장이 필요한가

부동산 데이터는 겉으로 보면 단순해 보인다. 실거래가 있으면 가격 흐름을 읽을 수 있고, 전세가율이 있으면 임차 시장을 볼 수 있고, 미분양이 있으면 공급 상황을 볼 수 있을 것처럼 보인다. 실제로 많은 분석은 이 수준에서 시작한다. 하지만 조금만 깊이 들어가면 문제가 바로 드러난다.

실거래가는 거래가 있었던 순간만 보여준다. 전세 데이터는 임차 시장의 상태를 보여주지만, 매매와 같은 속도로 반응하지 않는다. 입주예정 물량은 미래에 들어올 물량이지 현재 재고가 아니다. 미분양은 이미 소화되지 못한 재고이고, 준공후 미분양은 그중에서도 더 질이 나쁜 신호다. 정책은 발표 시점과 실제 반영 시점이 다를 수 있다. 상품성 데이터와 입지 데이터는 같은 지역 안에서도 단지별 차이를 설명해 주지만, 전국에 동일 품질로 깔려 있지 않다.

즉 부동산 데이터는 하나의 표로 정리되는 세계가 아니다. 역할이 다른 데이터들이 얽혀 있고, 그 데이터들을 어떤 순서와 계층으로 모으느냐에 따라 해석의 질이 크게 달라진다.

이 장의 목적은 바로 그 계층 구조를 세우는 것이다.

## 1. 부동산 데이터는 왜 쉽게 섞이는가

부동산 데이터가 자주 뒤섞이는 이유는 세 가지다.

### 1.1 시간축이 다르기 때문이다

- 실거래는 계약이 체결된 시점의 결과다.
- 정책은 발표 시점과 시행 시점이 다를 수 있다.
- 입주예정 물량은 미래 시점에 시장에 들어올 공급이다.
- 미분양은 이미 시장에 나와 있는데 흡수되지 않은 재고다.
- 호가 데이터는 실거래보다 먼저 움직일 수 있지만, 실제 체결 가격은 아니다.

이것들을 모두 같은 시간축에 놓으면 오류가 생긴다. 예를 들어 입주예정 물량이 많다는 사실을 현재 가격 약세의 직접 원인처럼 읽으면 안 된다. 반대로 미분양 감소를 곧바로 강세 신호로 읽는 것도 위험하다. 감소한 것이 일반 미분양인지, 준공후 미분양인지에 따라 의미가 달라지기 때문이다.

### 1.2 공간축이 다르기 때문이다

부동산은 전국 단위로도 읽고, 권역 단위로도 읽고, 시도 단위로도 읽고, 시군구 단위로도 읽고, 생활권 단위로도 읽고, 단지 단위로도 읽는다. 문제는 이 서로 다른 공간축이 같은 문장 안에서 섞이는 순간 판단이 흔들린다는 점이다.

예를 들어 `경기도가 강하다`는 말은 너무 넓고, `이 단지가 좋다`는 말은 너무 좁다. 실제 투자 판단은 그 사이 어딘가, 즉 `어느 생활권이 강한가`, `그 생활권 안에서 어떤 단지가 먼저 움직이는가`를 같이 봐야 한다.

### 1.3 데이터의 역할이 다르기 때문이다

같은 표 안에 있다고 해서 모두 같은 역할을 하는 데이터는 아니다.

- 실거래가, 전세, 월세는 `시장 반응` 데이터다.
- 공동주택 마스터는 `대상 식별` 데이터다.
- 정책 문서는 `해석 조건` 데이터다.
- 입주예정 물량, 미분양은 `공급과 재고` 데이터다.
- 좌표, 역세권, 학교, 상권, 정비사업 정보는 `설명력 보강` 데이터다.
- 네이버부동산 매물은 `현재 체감`을 보여주는 보조 데이터다.

이 역할을 구분하지 않으면, 본래 보조 지표여야 할 것이 주지표처럼 쓰이거나, 반대로 핵심 지표가 장식처럼 밀려나는 일이 생긴다.

## 2. 이 프로젝트에서 데이터를 나누는 기본 구조

이 프로젝트는 저장소 구조 자체가 데이터 계층 구조를 반영하도록 설계되어 있다.

### 2.1 기준 문서 층

`01_기준문서/`는 분석 기준과 실행 순서를 고정하는 층이다.

- 프롬프트
- 실행 가이드
- 설계 문서
- 체크리스트

이 층은 데이터를 담지 않는다. 대신 데이터를 어떻게 읽고, 어떤 순서로 작업하고, 어떤 표현을 써야 하는지를 정한다.

### 2.2 원천·참조 데이터 층

`02_데이터/`는 실제 분석에 쓰이는 원재료와 참조자료를 담는 층이다.

- 원천 데이터
- 참조표
- 가공 데이터
- 메모 자료

여기서 중요한 것은 모든 자료를 한 성격으로 보지 않는 것이다.

- `원천 데이터`는 아직 정리되지 않은 재료다.
- `참조 데이터`는 외부 기준표나 수동 정리표다.
- `가공 데이터`는 분석을 위해 한 번 정리한 결과다.
- `메모`는 데이터 해석을 보조하는 설명 문서다.

### 2.3 분석 실행 층

`03_분석/`은 스크립트, SQL, 뷰 정의, 임시 작업이 있는 층이다. 이 층에서 핵심 역할은 데이터를 설명하는 것이 아니라, 데이터를 실제 분석 가능한 구조로 만드는 것이다.

### 2.4 결과 층

`04_결과/`는 보고서, 방법론 문서, 노트북 산출물, 시각 자료가 쌓이는 층이다. 이 층은 데이터 생성 장소가 아니라 해석 결과가 남는 장소다.

즉 이 프로젝트는 처음부터 `원천 -> 정리 -> 분석 -> 산출` 흐름을 폴더 구조로 나눠두고 있다.

## 3. 책에서 독자에게 권하는 데이터 계층

독자는 자신의 환경에 맞게 조금씩 변형할 수 있지만, 최소한 아래 여섯 층은 분리하는 편이 좋다.

### 3.1 거래 데이터

시장 반응을 보여주는 중심 데이터다.

- 매매 실거래
- 전세 실거래
- 월세 실거래

이 층의 핵심 질문은 `실제로 얼마에 거래되었는가`다.

### 3.2 단지 마스터 데이터

거래 대상이 무엇인지 식별하는 층이다.

- 공동주택_전국
- 단지명
- 주소
- 건축년도
- 사용승인연도
- 세대수
- 동수
- 면적 구성

이 층의 핵심 질문은 `이 거래가 어떤 단지와 연결되는가`다.

### 3.3 정책 데이터

시장에 외부 충격을 주는 제약과 완충 조건을 설명하는 층이다.

- 대출 규제
- 세제 변화
- 토지거래허가구역
- 공급 확대 정책

이 층의 핵심 질문은 `이 시기와 지역에 어떤 정책 조건이 걸려 있었는가`다.

### 3.4 공급과 재고 데이터

가격이 아니라, 가격을 둘러싼 수급 여건을 설명하는 층이다.

- 입주예정 물량
- 미분양
- 준공후 미분양

이 층의 핵심 질문은 `앞으로 들어올 물량과 이미 쌓인 재고가 현재 해석을 어떻게 바꾸는가`다.

### 3.5 상품성/입지 데이터

같은 지역 안에서 왜 어떤 단지가 더 강한지 설명하는 보강 층이다.

- 연식
- 세대수
- 동수
- 면적 구성
- 역세권
- 학교/학원
- 상권
- 공원
- 한강/CBD 접근성
- 정비사업 정보

이 층의 핵심 질문은 `같은 생활권 안에서 단지별 차이를 설명하는 요인은 무엇인가`다.

### 3.6 현재 체감 보조 데이터

시장 분위기와 매물 압력을 읽는 보조 층이다.

- 네이버부동산_단지
- 네이버부동산_매물

이 층의 핵심 질문은 `실거래가 아직 완전히 반영되지 않은 현재 체감이 어떤가`다.

이 데이터는 중요하지만, 장기 패턴의 주지표로 쓰면 안 된다. 현재 체감과 보조 검증용으로 제한해야 한다.

## 4. 이 프로젝트가 실제로 택한 데이터 원칙

이 책은 추상적인 이상형을 말하는 데서 멈추지 않는다. 실제 프로젝트에서는 아래 원칙을 명시적으로 채택하고 있다.

### 4.1 핵심 DB는 외부 DuckDB 파일로 둔다

- 핵심 DB 경로: `/Volumes/T9/duckdb-analytics/db/apartment.duckdb`

저장소 안에 거대한 원천 DB를 넣지 않고, 외부 경로의 DuckDB를 중심 저장소로 둔다. 이렇게 하면 저장소는 기준 문서, 스크립트, 결과물 중심으로 가볍게 유지할 수 있다.

### 4.2 전국 아파트 마스터는 보수적으로 좁힌다

- `공동주택_전국`에서 `단지구분코드 = 1`만 사용한다.

즉 전국 분석에서 아파트 마스터의 기준을 명시적으로 제한한다. 이 기준이 흔들리면 뒤쪽의 단지 매칭과 상품성 해석이 함께 흔들린다.

### 4.3 서울 전용 데이터는 전국 공통 데이터처럼 쓰지 않는다

- `공동주택_기본정보`는 서울 데이터이므로 전국 기본 분석에서는 제외한다.
- `아파트_단지_위치변수.csv`는 수도권 심화 변수이므로 전국 공통 변수처럼 쓰지 않는다.

이 원칙은 매우 중요하다. 범위가 다른 데이터를 한 문장 안에 섞는 순간, 설명은 화려해져도 분석은 약해진다.

### 4.4 토지거래허가 데이터는 마스터로 다루지 않는다

DuckDB의 `토지거래허가구역` 테이블은 허가 승인과 수집 이력을 보여주는 보조 데이터다. 현재 활성 여부 자체는 최신 공식 공고와 수동 참조표로 다시 확인해야 한다.

즉 이 프로젝트는 `DB에 있으면 곧바로 진실`이라는 태도를 취하지 않는다. 이 태도가 데이터 구조 설계에서 중요하다.

## 5. 독자가 직접 구조를 세우는 순서

독자가 자신의 환경에서 같은 체계를 만들 때는 아래 순서로 가는 편이 좋다.

### 5.1 먼저 질문을 정한다

예를 들어 아래와 같이 질문을 먼저 정한다.

- 지금 강한 지역은 어디인가
- 아직 가격 반영이 덜 된 지역은 어디인가
- 입주예정 물량과 미분양은 어떤 지역에서 경고 신호인가

질문이 먼저 서야 데이터의 역할이 정해진다.

### 5.2 그다음 데이터의 역할을 정한다

각 질문마다 어떤 데이터가 중심이고 어떤 데이터가 보조인지 나눈다.

- 중심 데이터: 매매 실거래
- 보조 데이터: 전세가율, 입주예정 물량, 미분양, 정책, 호가

### 5.3 단위를 정한다

같은 데이터를 어떤 단위로 볼지 정한다.

- 권역
- 시도
- 시군구
- 생활권
- 단지

분석 단위가 흔들리면 결과도 흔들린다.

### 5.4 연결 키를 정한다

부동산 데이터는 결국 연결의 문제다.

- 지역 키
- 계약년월
- 면적 구분
- 단지 키
- 주소 키

이 키가 먼저 정리되어야 이후의 모든 해석이 가능해진다.

## 해석 예시

- 사실:
  이 프로젝트는 `거래`, `정책`, `공급과 재고`, `상품성/입지`, `현재 체감` 데이터를 분리해서 다룬다.
- 해석:
  이런 분리가 있어야 `왜 강한가`, `왜 아직 약한가`, `왜 같은 지역 안에서도 차이가 나는가`를 다른 층위에서 설명할 수 있다.
- 가설:
  독자가 자신의 환경에서도 같은 구조를 세우면, 개별 기사나 사례에 휘둘리는 빈도가 줄어들 가능성이 높다.

## 흔한 오해

- `부동산 데이터는 실거래만 있으면 충분하다`
  실거래는 중심 데이터지만, 그것만으로는 현재 체감과 공급·재고 상태를 충분히 설명하기 어렵다.
- `데이터는 많을수록 좋다`
  많기만 한 데이터보다 역할이 분리된 데이터가 더 중요하다.
- `전국 데이터와 수도권 심화 데이터를 한 번에 합치면 더 정교해진다`
  범위가 다른 데이터를 무리하게 합치면 정교해지는 것이 아니라 해석이 흔들린다.

## 직접 해볼 과제

1. 자신이 이미 갖고 있는 부동산 데이터를 모두 적어본다.
2. 그 데이터를 `거래`, `단지 마스터`, `정책`, `공급과 재고`, `상품성/입지`, `현재 체감`으로 다시 분류해본다.
3. 각 데이터마다 `이 데이터는 중심인가, 보조인가`를 표시해본다.
4. 같은 질문에 서로 다른 역할의 데이터가 섞여 있지 않은지 확인해본다.

## 체크포인트

- 나는 데이터를 종류가 아니라 역할 기준으로 나누고 있는가
- 나는 전국 공통 데이터와 수도권 심화 데이터를 분리하고 있는가
- 나는 중심 데이터와 보조 데이터를 같은 무게로 다루고 있지 않은가
